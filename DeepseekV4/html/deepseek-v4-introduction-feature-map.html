DeepSeek-V4 技术报告 · Introduction

Million-Token Context Intelligence

DeepSeek-V4 系列以高效百万上下文为主轴，将产品形态、架构创新、预训练、后训练和基础设施组织为一套面向长程任务与测试时扩展的系统方案。

1M上下文长度，支撑长程推理、跨文档分析和 agentic workflows。
32T+预训练 token 规模，覆盖数学、代码、网页、长文档等高质量类别。
27%V4-Pro 在 1M 上下文下相对 V3.2 的单 token FLOPs。
10%V4-Pro 在 1M 上下文下相对 V3.2 的 KV cache 体积。

产品形态：三种推理模式

DeepSeek-V4 最终面向使用时体现为三种 reasoning effort：Non-think、Think、Think Max；评测表中对应简写为 Non-Think、High、Max。三种模式同时覆盖 DeepSeek-V4-Pro 与 DeepSeek-V4-Flash 两条模型线。

Non-think评测表写作 Non-Think；快速、直觉式响应。 Think评测表写作 High；更充分的逻辑分析。 Think Max评测表写作 Max；最大推理努力。
DeepSeek-V4-Pro1.6T 总参数 / 49B 激活，偏能力上限。
Pro Non-think快答模式。
Pro Think评测列名为 Pro High。
Pro Think Max评测列名为 Pro Max；评测摘要多以 Pro-Max 呈现。
DeepSeek-V4-Flash284B 总参数 / 13B 激活，偏成本效率。
Flash Non-think低延迟响应。
Flash Think评测列名为 Flash High。
Flash Think Max评测列名为 Flash Max；更高 thinking budget 增强推理。
模式含义

三种模式不是三套架构，而是同一模型线在后训练中形成的不同推理努力配置。

模型对应

Pro 和 Flash 都支持三种模式；区别在于参数规模、激活规模、成本效率和能力上限。

Posttrain：专家能力的统一合成

后训练的完整链路是：从 Base Model 出发，先训练多个领域专家，再补充推理模式、奖励建模和工具交互能力，最后通过 full-vocabulary OPD 合并为统一模型。

Base Model 预训练后的 Flash-Base / Pro-Base，已经具备 1M 上下文基础能力。
来自 Pre-Training
→
1. 领域专家训练 按数学、代码、Agent、指令遵循等领域分别训练专家。
SFT 建基础解法；GRPO 学习领域 reward 下更优的回答轨迹。
Reasoning Effort：为 Non-think / Think / Think Max 训练不同推理预算。
GRM：处理 hard-to-verify 任务，用生成式评审给奖励。
→
2. Agent 与交互能力 让模型在工具调用和多轮任务中保持状态，并降低辅助任务延迟。
Interleaved Thinking：工具场景保留跨轮 reasoning history。
Quick Instruction：用特殊 token 复用 KV cache 执行搜索触发等辅助任务。
→
3. OPD 合并 用多个专家作为 teacher，让统一 student 在自身轨迹上对齐 full-vocabulary logits。
输出：统一最终模型 + 三种推理模式 + benchmark / real-world tasks 验证。

Posttrain 思路对比：V3 / GLM-5 / V4

三份技术报告的分水岭在于：V3 侧重“把强推理能力注入通用模型”，GLM-5 侧重“连续异步 RL 训练长程 Agent”，V4 侧重“多专家能力通过 OPD 合并成统一模型”。

DeepSeek-V3

后训练是经典 SFT + RL：SFT 阶段从 DeepSeek-R1 系列蒸馏推理数据，再用 GRPO 结合 rule-based / model-based reward 做通用对齐。

核心目标是把 reasoning 能力注入 V3，同时保持回答长度、准确性和通用偏好之间的平衡。

关键词：R1 distillation → SFT → GRPO RL。

GLM-5

后训练强调顺序式 RL pipeline：Reasoning RL → Agentic RL → General RL，并用 asynchronous RL infrastructure 将 rollout generation 与 model update 解耦。

核心目标是让模型持续学习长程 Agent 交互，尤其提升复杂软件工程任务中的规划、自我修正和工具使用。

关键词：sequential RL + async agent RL + cross-stage distillation。

DeepSeek-V4

后训练先独立训练数学、代码、Agent、指令遵循等领域专家，再用 multi-teacher On-Policy Distillation 将多个教师能力整合进一个学生模型。

核心变化是用 OPD 替代混合 RL 合并阶段，避免专家权重或能力直接混合带来的退化，并用 full-vocabulary logits 提高蒸馏稳定性。

关键词：specialists → full-vocab OPD → unified model。

Architecture：百万上下文的计算基础

第 2 章对应 Introduction 的核心技术问题：在降低长序列注意力成本的同时，保持模型表达力、训练稳定性和推理效率。

DeepSeekMoE + MTP 继承

沿用 MoE 和 Multi-Token Prediction，保留 V3 已验证的高效能力底座。

对应问题：能力底座复用

mHC 残差连接

把残差映射约束到双随机矩阵流形，控制谱范数，增强层间信号传播稳定性。

对应问题：深层堆叠数值不稳

CSA 压缩稀疏注意力

先把每 m 个 KV 压成一个 entry，再用 DSA 只选择 top-k 压缩 KV 参与核心注意力。

对应问题：长序列 FLOPs 激增

HCA 重压缩注意力

以更大压缩比 m′ 聚合 KV，保留 dense attention，用极致压缩降低 KV cache 规模。

对应问题：1M KV 存储不可承受

SWA + Attention Sink

滑动窗口补回局部依赖，attention sink 允许头部调节总注意力质量。

对应问题：压缩后局部细节损失

Muon Optimizer

大多数模块用 Muon，配合 hybrid Newton-Schulz 正交化，提升收敛速度和稳定性。

对应问题：大模型训练收敛与稳定

Pretrain：架构、数据与训练策略汇合

第 4 章将 32T+ token、原生 1M 上下文和 base model 能力提升落到数据构造、长度课程和稳定性治理上。

长文档与高质量语料

在 V3 数据上强化多样性，过滤自动生成/模板内容，突出论文、技术报告等长文档。

对应问题：长上下文缺少高质量来源

代码、数学、Agentic Data

数学和编程仍是核心语料，并在 mid-training 引入 agentic data 强化代码与工具任务。

对应问题：推理与 Agent 能力缺口

4K → 16K → 64K → 1M

训练从短序列逐步扩展到百万长度，让模型逐渐适应超长上下文。

对应问题：直接长训不稳定/成本高

Dense Warmup → Sparse Attention

先用 dense attention warmup，再在 64K 后引入 sparse attention，并短阶段预热 lightning indexer。

对应问题：稀疏注意力冷启动

Anticipatory Routing

使用历史参数预先计算 routing indices，在 loss spike 时短暂启用，切断路由导致的异常循环。

对应问题：MoE outlier 与 loss spike

SwiGLU Clamping

限制 SwiGLU 的线性和 gate 分量，直接压制异常值而不牺牲性能。

对应问题：训练中数值异常

Infra：规模化训练与服务底座

第 3 章的重点不是单个工程技巧，而是三层基础设施能力：让 MoE 通信不拖慢计算，让新算子高效且可复现，让训练、推理和 Agent 评测在百万上下文下可持续运行。

MoE 执行层 Expert waves 将 Dispatch / GEMM / Combine 串成流水线，把跨节点通信隐藏在专家计算中。
Kernel 可信层 TileLang + deterministic kernels 让新算子既能快速开发，又能在训练、后训练和推理之间保持 bitwise 对齐。
状态管理层 FP4、Hybrid KV cache、on-disk KV 与 DSec 管理显存、缓存、rollout 和 sandbox 生命周期。

1. 并行通信隐藏

针对 MoE Expert Parallelism 的跨节点通信瓶颈，使用 expert waves 与 fused mega-kernel 将 Dispatch、GEMM、Combine 组织成细粒度流水线，把通信藏在计算下面。

MegaMoEEP overlap1.50-1.96× speedup

2. Kernel 与可复现体系

针对 CSA/HCA/mHC 等新结构带来的大量细粒度算子与调试困难，使用 TileLang 融合 kernel、Host Codegen、batch-invariant 与 deterministic kernel，兼顾开发效率、性能和 bitwise 对齐。

TileLangdeterministic kernelsbitwise alignment

3. 训练与推理状态管理

针对百万上下文带来的显存、KV cache、rollout 和 Agent sandbox 压力，使用 FP4 QAT、Muon/mHC/CP 训练实现、异构 KV cache、on-disk prefix reuse 与 DSec，支撑长上下文训练、服务和评测。

FP4 QATHybrid KV cacheDSec sandbox
Posttrain · Specialist Training

领域专家 SFT + RL

针对统一模型难以在数学、代码、Agent、指令遵循中同时专精的问题，先用领域 SFT 教会基本解法，再用 GRPO 学习“同一题里哪些回答相对更符合领域 reward”，得到可被后续蒸馏合并的专家能力。

GRPO 学什么

它不是直接学习标准答案文本，而是在一组候选回答之间比较相对优劣：哪个推理更完整、哪个代码能通过测试、哪个工具调用轨迹更符合目标 reward。

例子：代码专家

同一道编程题采样 4 个解法：A 编译失败，B 只过样例，C 通过全部单元测试但解释混乱，D 通过测试且边界处理清楚。GRPO 根据测试结果和代码质量 reward 提升 D/C 这类轨迹概率，压低 A/B。

例子

Reward signal若摘要准确覆盖事实且建议谨慎，给高奖励；若漏掉违约条款或语气武断，给低奖励。

Posttrain · Multi-Teacher Merge

Full-Vocabulary OPD

针对多个专家模型直接合并容易退化的问题，使用学生在自己生成的轨迹上，对教师的完整词表概率分布做 reverse KL 对齐，得到更稳定的多教师能力合并。

方法怎么做

先让学生模型自己生成回答轨迹；再让相关教师模型在这些位置上输出完整 vocabulary logits；最后让学生分布靠近教师分布，而不是只模仿已经采样出来的那个 token。

为什么叫 full-vocabulary

因为它看的是每个位置上所有候选词的概率形状。比如教师不仅认为“therefore”好，也认为“thus”接近可用、而“banana”极差；这些暗知识都会传给学生。

例子

数学专家看到学生写到“所以答案是”，完整词表会给 “42” 高概率、给 “因此” 中等概率、给无关词低概率；学生学到的是这一整个偏好分布，而不只是硬抄一个 “42”。

分布示意
42thereforethusbanana
Teacher0.620.180.140.00
Student0.310.350.080.04

训练目标是把 Student 的整行分布拉向 Teacher，而不是只奖励某一个采样 token。

Architecture · Residual Path

mHC 残差连接

ResNet：单一路径残差

x → F(x) → x + F(x) ↷ skip x

普通 ResNet 只有一条 residual stream：层输出直接加回原输入，结构简单但扩展维度有限。

Hyper-Connected / mHC：多 residual stream

x₁x₂x₃x₄
A 选择层输入 → F(A·X) → C 写回；B 在多条 stream 间混合
x′₁x′₂x′₃x′₄

Hyper-Connected ResNet 扩展为多条 residual stream；mHC 进一步把 B 约束为双随机矩阵，让混合非扩张，避免深层堆叠不稳定。

针对 Hyper-Connections 深层堆叠容易数值不稳的问题，mHC 不再只用单一路径残差，而是在多条 residual stream 之间做受约束混合：A 选择层输入、C 写回输出、B 负责跨 stream 混合，并把 B 约束到双随机矩阵流形以稳定传播。

Architecture · Hybrid Attention

CSA / HCA 混合注意力

针对百万上下文中注意力 FLOPs 和 KV cache 同时膨胀的问题，CSA 用“小段压缩 + 稀疏选择”降低计算，HCA 用“长段重压缩”降低缓存，二者交替使用让 1M context 可行。

CSA：短块压缩后再稀疏选择

t1t2t3t4→C1
t5t6t7t8→C2

每 m 个 KV token 压成一个 compressed KV entry；查询 token 不看全部 C，而是由 lightning indexer 选择 top-k 个压缩块参与注意力。

HCA：更长块直接重压缩

t1t2t3t4t5t6t7t8→H1

每 m′ 个 KV token 压成一个 HCA entry，且 m′ 远大于 m；它牺牲更细粒度索引，换取更小 KV cache 和更低长上下文存储压力。
